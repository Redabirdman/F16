# F16 — V1 System Design

**Project:** F16 (internal codename) — Autonomous AI organization for Assuryal Conseil insurance reselling
**Owners:** Ridaa (AI, infra), Achraf (insurance, product, growth)
**Date:** 2026-05-17
**Status:** Design — awaiting Ridaa + Achraf approval before implementation plan

---

## 1. Executive Summary

F16 replaces the human call-agent team that today fields Meta-form leads for Assuryal Conseil (car and electric-scooter insurance in France). A multi-agent AI organization runs the full path from ad creative to signed contract handoff, operating 24/7 over WhatsApp, voice, email, and SMS. Ridaa and Achraf step in for funding, ad approvals, wire-transfer / contract issuance with Maxance, and any escalation flagged by the system.

**V1 ships complete.** Not a slice. Sixteen agent roles, four customer channels, a 2D isometric live "office" visualization plus a working admin panel, all hosted on Ridaa's existing Hostinger VPS via EasyPanel.

**Voice is in V1.** Reversed from earlier deferral after the Dograh / Pipecat / OVH-SIP investigation showed per-minute cost of €0.04–0.07 vs Vapi's €0.15–0.20.

**Estimated monthly running cost at ~50 leads/day:** €800–1,100 across all LLM, voice, infra, and integration fees combined.

## 2. Business Context

### 2.1 Today (without F16)

- Meta ads (FB + Instagram) → HubSpot lead form → lead lands in Achraf's website Supabase + HubSpot
- Human call agents phone the lead
- Agent gathers vehicle / driver info while logging into Maxance (web app of underwriter)
- Maxance shapes a price; agent downloads / emails the quote
- Agent re-engages over phone or WhatsApp to close
- On acceptance: agent re-opens Maxance, collects banking info, gives wire instructions
- Customer wires payment; agent calls Maxance human to issue the actual contract
- Agent emails the contract to the customer

**Pain:** large fraction of leads never reach the close because customers don't answer the call, or the agent reaches them but loses the thread in follow-up.

### 2.2 With F16

The full path above runs autonomously across WhatsApp (primary), voice (callback for silent leads + inbound "call me"), email, and SMS. Maxance is operated by a browser agent. Memory of every customer is permanent. Ads are designed, posted, and optimized continuously, with humans approving budget and creative direction. A daily Reporter Agent talks to Ridaa + Achraf in their WhatsApp group. The wire-transfer and Maxance contract-issuance calls remain 100% human.

### 2.3 Brand

- **F16** — internal codename. Never seen by customers.
- **Assuryal** — customer-facing brand. All agents speak as Assuryal. Site: assuryalconseil.fr.
- **AW Assur Conseil** — separate brand under same operators; F16 inherits creative-pipeline patterns from it but is not branded as it.

## 5. Agent Catalog

Sixteen roles total: 5 persistent specialists + 9 ephemeral workers + 2 supervisor faces (one entity).

### 5.1 Persistent specialists

#### Sales Agent

- **Instances:** one per active customer thread (lead.status in {NEW, QUALIFYING, QUOTING, NEGOTIATING, AWAITING_PAYMENT})
- **Model:** Claude Sonnet 4.6 with prompt caching
- **Owns:** the entire conversation with one customer across any channel. Qualifies, gathers info, requests quotes from Maxance Operator, delivers them, closes, hands off to humans for wire transfer.
- **Tools:** `customer.read_profile`, `customer.update_profile`, `quote.request`, `quote.send_pdf`, `human.escalate`, `channel.switch_request`, `knowledge.search`, `memory.recall_customer`
- **Termination:** thread reaches CLOSED*WON, CLOSED_LOST, or DORMANT*>30d

#### Maxance Operator

- **Instances:** one per active quote session (concurrency-limited by Stagehand container count; V1 = 3 parallel)
- **Model:** Claude Sonnet 4.6 with vision (for screenshot interpretation)
- **Owns:** logged-in Maxance session for one quote. Navigates the multi-step quote flow (Véhicule → Conducteur → Garanties → Infos complémentaires → Coordonnées souscripteur), fills fields from the Sales Agent's structured payload, captures resulting price, downloads/emails quote PDF.
- **Tools:** Stagehand intent API, `screenshot.capture`, `quote.return_to_sales`, `human.escalate` (when Maxance UI behavior is unfamiliar)
- **Termination:** quote delivered or session expires after N minutes idle

#### Ads Manager

- **Instances:** singleton, continuous-loop
- **Model:** Claude Sonnet 4.6
- **Owns:** Meta campaigns. Polls performance metrics, detects fatigue (CTR drop, frequency creep), requests new creatives from Creative Brief Writer, queues budget changes for human approval, posts approved creatives, requests lookalike-audience refresh weekly.
- **Tools:** Meta Marketing API (read metrics, create/update campaigns/adsets/ads), `creative.request_brief`, `human.request_approval`, `audience.refresh_lookalike`
- **Loop cadence:** every 15 min for metrics; on-demand for fatigue

#### Customer Engagement

- **Instances:** singleton, continuous-loop
- **Model:** Claude Sonnet 4.6
- **Owns:** follow-up cascades for cold or silent leads. Decides which lead is due for which touch on which channel, drafts via the Follow-up Composer ephemeral, dispatches via channel.
- **Tools:** `lead.find_due_followups`, `composer.draft_followup`, `channel.send`, `voice.schedule_callback`
- **Loop cadence:** every 5 min

#### Knowledge Curator

- **Instances:** singleton, low-frequency (every 6h + on-demand)
- **Model:** Claude Haiku 4.5
- **Owns:** the F16 knowledge store. Re-ingests assuryalconseil.fr and Maxance product pages, chunks, embeds, indexes into pgvector. Diffs against previous snapshot — surfaces drift (price changes, new product, new exclusions) to the Supervisor.
- **Tools:** website scraper, Maxance browser (read-only), `knowledge.upsert_chunk`, `drift.report`

### 5.2 Ephemeral workers (BullMQ-spawned, die after one task)

| Role                  | Model                      | Triggered by                                                                  | Returns                                                          |
| --------------------- | -------------------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Creative Brief Writer | Sonnet 4.6                 | Ads Manager → `CREATIVE.BRIEF_REQUESTED`                                      | French ad-copy + creative prompt (psychology angle + brand spec) |
| Creative Generator    | none (Python subprocess)   | Brief Writer                                                                  | PNG paths                                                        |
| Lead Scorer           | Haiku 4.5                  | `LEAD.NEW`                                                                    | intent score 0-100 + best opening message + recommended channel  |
| Quote Summarizer      | Haiku 4.5                  | `QUOTE.READY`                                                                 | French WhatsApp message + email body                             |
| Follow-up Composer    | Sonnet 4.6                 | Customer Engagement                                                           | next French message in cascade, with channel recommendation      |
| OCR Worker            | Claude Vision (or chandra) | `CUSTOMER.OCR_REQUESTED`                                                      | structured JSON of carte grise / permis fields                   |
| Audience Builder      | none (Meta API)            | Ads Manager → `AUDIENCE.REFRESH_REQUESTED`                                    | new lookalike audience ID                                        |
| Email Sender          | none (BillionMail wrapper) | any agent → `EMAIL.SEND_REQUESTED`                                            | message ID + delivery proof                                      |
| Compliance Sentry     | Haiku 4.5                  | every Sales Agent outbound draft → `COMPLIANCE.CHECK_REQUESTED` (synchronous) | PASS / BLOCK + reason                                            |

### 5.3 Supervisor (one singleton entity, two faces)

#### Team Manager (internal face)

- **Model:** Opus 4.7 for daily / hourly strategy reviews; Sonnet for routine routing
- **Owns:** inter-agent escalations, resource priorities, kill switches, agent registry / topology
- **Tools:** full read of `agent_messages`, `agent.spawn`, `agent.kill`, `agent.adjust_priority`, `human.escalate`, `audit.read`

#### Reporter (external face)

- **Model:** Opus 4.7 for daily digest; Sonnet for action notices
- **Owns:** the WhatsApp group with Ridaa + Achraf, plus `/team-chat` in the admin panel (mirror). Sends daily KPI summary, action requests, urgent escalations. Receives human resolutions and applies them.
- **Tools:** WhatsApp group send/receive, `human_action.list_pending`, `human_action.resolve`, KPI snapshot tools

## 6. Inter-Agent Communication Protocol

### 6.1 The pattern

All agent-to-agent communication is a typed event persisted to Postgres `agent_messages`, dispatched by BullMQ. **Persisted** because:

1. Full audit trail for the admin panel (`/leads/:id` timeline pulls these directly)
2. Crash recovery (in-flight messages survive process restart)
3. Replay / debugging
4. Realtime fan-out to office visualization via Postgres LISTEN/NOTIFY

### 6.2 Schema

```sql
CREATE TABLE agent_messages (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  from_role       TEXT NOT NULL,
  from_instance   TEXT,          -- "sales_agent#lead-1234", null for singletons
  to_role         TEXT NOT NULL, -- can be 'supervisor', 'human', 'sales_agent', ...
  to_instance     TEXT,          -- if null = any instance of to_role
  intent          TEXT NOT NULL, -- e.g. 'QUOTE.REQUESTED'
  payload         JSONB NOT NULL,
  correlation_id  TEXT,          -- lead_id, campaign_id, customer_id
  requires_human  BOOLEAN NOT NULL DEFAULT false,
  priority        SMALLINT NOT NULL DEFAULT 5, -- 0=critical .. 9=background
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  consumed_at     TIMESTAMPTZ,
  consumed_by     TEXT,
  result          JSONB,
  error           TEXT
);
CREATE INDEX ON agent_messages (to_role, consumed_at) WHERE consumed_at IS NULL;
CREATE INDEX ON agent_messages (correlation_id);
CREATE INDEX ON agent_messages (created_at);
```

Every intent has a zod schema validating `payload`. Producing or consuming an intent without a registered schema is a runtime error (caught in CI).

### 6.3 Intent enum (V1 core)

Organized by domain. New intents added by config + zod schema — no orchestrator code change.

## 7. Memory Architecture

Three tiers, all on Postgres + pgvector. Mem0 as a recall-API facade so agents never write SQL directly.

### 7.1 Customer memory

Every fact known about one customer.

```sql
CREATE TABLE customers (
  id              UUID PRIMARY KEY,
  full_name       TEXT, dob DATE, civility TEXT,
  email           TEXT, phone TEXT, address JSONB,
  vehicle         JSONB,   -- type, make, model, year, plate, etc.
  driver          JSONB,   -- license type, points, malus, alcohol history, etc.
  preferences     JSONB,   -- preferred channel, time-of-day, language nuance
  iban_hash       TEXT,    -- IBAN itself encrypted at rest, hash for dedup
  hubspot_id      TEXT,
  created_at      TIMESTAMPTZ, updated_at TIMESTAMPTZ,
  consent         JSONB    -- GDPR record: when, what, opted-out flags
);

CREATE TABLE customer_facts (
  id              UUID PRIMARY KEY,
  customer_id     UUID REFERENCES customers,
  fact_type       TEXT,    -- 'objection', 'preference', 'observation', 'event'
  content         TEXT,
  confidence      REAL,
  recorded_by     TEXT,    -- which agent
  recorded_at     TIMESTAMPTZ,
  embedding       VECTOR(1536)
);

CREATE TABLE conversation_turns (
  id              UUID PRIMARY KEY,
  customer_id     UUID,
  channel         TEXT,    -- 'whatsapp' | 'voice' | 'email' | 'sms'
  direction       TEXT,    -- 'inbound' | 'outbound'
  agent_role      TEXT,
  agent_instance  TEXT,
  content         TEXT,
  attachments     JSONB,   -- urls to images/audio/pdfs
  occurred_at     TIMESTAMPTZ,
  embedding       VECTOR(1536)
);
```

Recall API:

```ts
mem0.recallCustomer({ customerId, query, limit }); // semantic search over facts + turns
mem0.upsertFact({ customerId, fact, type, confidence, agent });
```

### 7.2 Per-agent-role memory (learned patterns)

```sql
CREATE TABLE agent_patterns (
  id              UUID PRIMARY KEY,
  agent_role      TEXT,
  pattern_type    TEXT,    -- 'heuristic', 'objection-handler', 'opener'
  trigger_summary TEXT,    -- "lead asks 'c'est cher'"
  recommended_action TEXT, -- "reframe as café/week — converts 28% vs 12%"
  evidence_count  INT,
  win_rate        REAL,
  last_used_at    TIMESTAMPTZ,
  embedding       VECTOR(1536)
);
```

Patterns are written by the agents themselves after `QUOTE.ACCEPTED` / `QUOTE.REJECTED` events (Opus 4.7 weekly distillation job). Sales Agent retrieves the top-N patterns matching the current customer state on each turn.

### 7.3 Org-wide memory (KPIs + decisions)

Standard OLTP tables: `campaigns`, `ads`, `creatives`, `ad_metrics_hourly`, `kpi_daily`, `human_actions`, `audit_log`. Time-series rolled up nightly. Powers `/dashboard`.

## 8. Channel Abstraction

```ts
interface ConversationChannel {
  id: 'whatsapp' | 'voice' | 'email' | 'sms';
  send(opts: {
    to: ContactRef;
    body: ContentBlock[];
    replyTo?: MessageRef;
  }): Promise<DeliveryReceipt>;
  capabilities(): { interactive: boolean; voice: boolean; attachments: boolean; markdown: boolean };
}
```

Every channel implements the same interface. Agents never speak to WAHA or Pipecat or BillionMail directly — they speak to the channel layer.

Implementations in V1:

| Channel    | Backend                                                           |
| ---------- | ----------------------------------------------------------------- |
| `whatsapp` | WAHA HTTP API (existing on VPS)                                   |
| `voice`    | Pipecat HTTP bridge (Python service on VPS) → OVH SIP trunk       |
| `email`    | BillionMail SMTP wrapper                                          |
| `sms`      | android-sms-gateway (only as fallback when WhatsApp + voice fail) |

V2 channels (e.g., Telegram, web chat widget, voice cloning) plug in by implementing the interface and registering with the channel layer.

### 8.1 Channel switching policy

The Sales Agent (not Customer Engagement) decides switches mid-conversation:

- Default: stay on the channel the customer started on.
- Customer says "appelez-moi" → schedule voice callback.
- Customer goes silent on WhatsApp > 24h → Customer Engagement Agent escalates to voice after one more WhatsApp ping.
- Voice call fails twice consecutively → Customer Engagement fall back to SMS.

## 9. Maxance Integration

### 9.1 Approach

Stagehand wraps Playwright with LLM-driven intents. The Maxance Operator agent speaks in intents:

```
"Open Maxance in a logged-in session"
"Navigate to Tarif > Nouveau Client"
"Set type véhicule to 'Trottinette électrique'"
"Capture the current displayed price"
"Click Coordonnées souscripteur, fill subscriber form with these fields: ..."
"Download the quote PDF for this devis"
```

Stagehand resolves these to DOM actions at runtime using Claude vision + DOM. UI changes survive without code changes.

### 9.2 Container

`f16-stagehand` EasyPanel container — headless Chromium, persistent user-data dir for login cookies, exposes an HTTP API to the F16 backend. Concurrency limit V1: 3 parallel quote sessions (raise as load demands).

### 9.3 Session lifecycle

- One long-lived login. SESSION.HEARTBEAT pings every 5 min; SESSION.LOGGED_OUT triggers re-login via stored MFA recovery flow (manual if Maxance has 2FA — falls to human via HUMAN_ACTION.REQUESTED).
- Per-quote sessions are sub-sessions inside the logged-in browser.
- Every step persisted: action description + before/after screenshot URLs in `maxance_actions` table.

### 9.4 Proof-of-action

Every Maxance step writes a row:

```sql
CREATE TABLE maxance_actions (
  id              UUID PRIMARY KEY,
  customer_id     UUID,
  quote_id        UUID,
  action          TEXT,           -- the intent text
  step_index      INT,
  screenshot_before_url TEXT,
  screenshot_after_url TEXT,
  duration_ms     INT,
  result          JSONB,
  error           TEXT,
  occurred_at     TIMESTAMPTZ
);
```

`/leads/:id` admin route renders these in a vertical timeline alongside conversation turns — full audit per customer.

## 11. Voice Stack

### 11.1 Components

```
                  Customer phone
                       │
                       ▼
                 OVH SIP trunk France
                       │
                       ▼
            ┌──────────────────┐
            │ Pipecat container │  (Python, EasyPanel)
            │                  │
            │  • Audio I/O     │
            │  • Deepgram STT  │── French streaming
            │  • Azure TTS     │── Henri / Denise voices
            │  • Turn-taking   │
            │  • VAD           │
            │                  │
            └─────────┬────────┘
                      │ HTTP POST /v1/sales-agent/turn
                      ▼
            F16 Backend (TypeScript) ── Sales Agent brain
                      │
                      ▼
              text response → TTS → audio
```

### 11.2 Why this layout

- Brain stays in TypeScript. No prompt duplication.
- Voice and WhatsApp converge on the same `conversation_turns` rows; same memory; same Sales Agent personality. A customer switching from WA to voice mid-funnel doesn't lose state.
- TTS swappable per customer or per call. Default Azure French (cheap, good); upgrade to ElevenLabs for high-value leads if we want.

### 11.3 V1 voice flows

1. **Outbound callback** — Customer Engagement decides a silent WhatsApp lead needs voice. Schedules a Pipecat outbound call via BullMQ.

## 13. Human-Action Channel (dual-surface approvals)

Critical V1 design: every item that needs human resolution exists in BOTH surfaces, and either resolves it.

### 13.1 Lifecycle

```
Any agent → HUMAN_ACTION.REQUESTED
               │
               ▼
        human_actions row inserted
        (status=PENDING, due_at, severity, summary, options[])
               │
               ├──▶ Admin /queue/human row appears (realtime)
               │
               └──▶ Reporter Agent posts to WhatsApp group:
                       "⚠️ [HA-1234] New action needed — approve creative V2_Bonus_PrixBas_1x1.
                        Reply 1 = approve, 2 = revise, 3 = reject. Or open admin."
                       (interactive buttons when WAHA supports; numbered fallback always)
                          │
                          ▼
        Resolution from either side hits the same handler:
        human_action_resolve(action_id, choice, by_user, source) →
            HUMAN_ACTION.RESOLVED → original agent picks up
               │
               ├──▶ Admin row updates (✅ Approved by Ridaa, 14:32, via WhatsApp)
               └──▶ WhatsApp thread replied (✅ Done)
```

### 13.2 Auth

Only WA messages from Ridaa's or Achraf's whitelisted numbers count. Anyone else in the group is ignored for resolution (still visible, just inert).

### 13.3 Severity tiers

| Tier        | Examples                                                                              | SLA                                    |
| ----------- | ------------------------------------------------------------------------------------- | -------------------------------------- |
| 🔴 Critical | refund, complaint, legal threat, Maxance session locked out, payment-pending mismatch | <15 min ping + audible WA notification |
| 🟡 Standard | new creative approval, budget change, ambiguous customer ask                          | within 4h                              |
| 🟢 Info     | daily Reporter digest, knowledge-drift notice                                         | no SLA                                 |

## 17. Security & Compliance

### 17.1 Operational guardrails (approved 2026-05-16)

1. Agents never claim a contract is bound before Maxance human-side completes.
2. Full conversation logs persisted as proof in each customer record.
3. Auto-handoff to Ridaa/Achraf for refund disputes, contract terminations, complaints, or any out-of-policy ask.

Enforced by Compliance Sentry on every Sales Agent outbound + by explicit `HUMAN_ACTION.REQUESTED` rules in the agent prompts + by a server-side rule check before message dispatch (defense in depth).

### 17.2 Data protection

- All customer PII column-encrypted at rest
- TLS everywhere (Caddy + internal mTLS between containers)
- WAHA session tokens in EasyPanel secrets, never in DB
- Stagehand login cookies on encrypted volume, never logged
- Customer data export endpoint for GDPR Subject Access Requests (V1 simple JSON)
- Customer data delete endpoint for GDPR erasure (V1 cascade by `customer_id`)
- Retention defaults: 7 years for closed contracts (French insurance norms; Achraf to confirm), 90 days post-rejection for cold leads

### 17.3 Auth

- Admin panel: WebAuthn + email magic link, 2FA mandatory for Ridaa and Achraf
- HMAC on the public lead-intake webhook from the website
- WhatsApp group resolutions only honored from whitelisted numbers (Ridaa, Achraf)

### 17.4 Performance / Speed SLOs (Ridaa mandate, 2026-05-17)

F16's value vs human call agents is speed. SLOs the system must hold:

| Flow                                                             | Target                        | Hard ceiling |
| ---------------------------------------------------------------- | ----------------------------- | ------------ |
| Lead-form submit → first WhatsApp opener visible to customer     | **< 30 sec**                  | 60 sec       |
| WhatsApp inbound → Sales Agent first token (streaming)           | **< 3 sec**                   | 6 sec        |
| WhatsApp inbound → Sales Agent full reply sent                   | **< 8 sec**                   | 15 sec       |
| Voice answer (inbound) → first agent audio                       | **< 800 ms**                  | 1.5 sec      |
| Customer accepts quote terms → Maxance form pre-fill begins      | **< 1 sec** (already running) | 3 sec        |
| Customer provides full info → Maxance quote PDF in customer hand | **< 3 min**                   | 5 min        |
| Lead arrives → quote delivered (happy path end-to-end)           | **< 5 min**                   | 10 min       |
| Human-action request → WhatsApp group notification               | **< 5 sec**                   | 10 sec       |

**Engineering levers (must be present in implementation plan):**

1. **Streaming text** to WhatsApp "typing…" from the first Sonnet token, not after completion.
2. **Speculative parallelism**: Lead Scorer + Sales Agent both start on `LEAD.NEW`; Sales Agent sends opener immediately, scoring informs turn 2.
3. **Pre-warm Maxance Operator**: Stagehand opens the quote form in background as soon as Sales Agent has the customer's vehicle category — _before_ the customer says "yes give me a quote".
4. **Inline Compliance Sentry on short messages only** (sub-300ms Haiku); long messages get async + retroactive check.
5. **Aggressive prompt caching** for every Claude SDK call.
6. **Always-warm Stagehand browser pool** with maintained Maxance login (no cold-start).
7. **Streaming STT/TTS** in voice + partial-result TTS playback (mask LLM latency under speech).
8. **Co-located containers**: Pipecat + WAHA + backend on same VPS to keep round-trip sub-50ms within the rack.
9. **No synchronous polling on customer-visible paths** — everything event-driven via BullMQ + Postgres LISTEN/NOTIFY.

Latency is monitored per-flow and dashboarded in `/dashboard` as a first-class KPI alongside conversion rate.
