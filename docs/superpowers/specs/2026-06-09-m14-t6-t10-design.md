## T6 — Generic prompt-override system + admin editor

### Goal

Let Ridaa/Achraf edit **any agent's prompt** from the admin; the agent's **next
message uses the edited prompt**; every change is audited and reversible.
Decision: **every agent, generically**, via a **registry + whole-prompt override**.

### Architecture

- **Registry** `backend/src/prompts/registry.ts`:
  - `registerPrompt({ key, label, agentRole, description, getDefault })` — called at
    module load by each prompt site. `key` is a stable dotted id
    (e.g. `sales-agent.system`, `voice.persona`, `reporter.interpret`,
    `creative.generate`, `creative.learnings-distill`, `creative.brand`,
    `lead-scorer.score`, `compliance.sentry`, `supervisor.strategy`,
    `engagement.nudge`).
  - `listPrompts()` → registry metadata (key, label, agentRole, description, default).
  - `resolvePrompt(db, key, getDefault)` → returns the DB override `content` if a
    row exists for `key`, else `getDefault()`. **Always falls back to the code
    default** on any DB/cache error (an agent can never be broken by this).
  - In-memory override cache with a monotonically-bumped version; `bustCache()` on
    every write. (Single backend process on this PC — no cross-process invalidation
    needed.)
- **DB** migration: table `prompt_overrides (key text PRIMARY KEY, content text NOT
NULL, updated_by text, updated_at timestamptz default now())`. Repo
  `backend/src/db/repositories/prompt-overrides.ts`: `getOverride(key)`,
  `listOverrides()`, `upsertOverride(key, content, updatedBy)`, `deleteOverride(key)`.
- **Wiring** — each prompt site changes from building a string inline to:
  `const sys = await resolvePrompt(db, '<key>', () => <existing build>)` and a
  one-time `registerPrompt({...})`. "Whole-prompt" = each key maps to ONE editable
  system-prompt string; per-turn dynamic context is appended AFTER (unchanged).
  - **Sales Agent:** register the STABLE prefix (brand+products+playbook+guardrails
    assembled by `buildSalesAgentSystemPrompt`). The per-turn context + Anthropic
    prompt-cache breakpoint stay after it; an override changes the cached prefix
    text → one cache miss, then re-cached. Acceptable.
  - **Voice persona** (`http/voice-persona.ts`): register the persona string.
  - Others (reporter interpret, creative generate/learnings/brand, lead-scorer,
    compliance sentry, supervisor strategy, engagement nudge): wrap their built
    system string under a key.
- **Admin endpoints** `backend/src/admin/prompts.ts`:
  - `GET /v1/admin/prompts` → `[{ key, label, agentRole, description, default,
override|null, isOverridden }]` (merge registry + overrides).
  - `PUT /v1/admin/prompts/:key` `{ content }` → upsert + `audit_log` +
    `bustCache()`. 404 if key not in registry; reject empty content.
  - `DELETE /v1/admin/prompts/:key` → delete override (reset to default) + audit +
    bust. Idempotent.
  - Mounted in `index.ts`; behind the existing admin bearer-auth.

### Frontend

- `admin/src/pages/Prompts.tsx` → `/prompts` (nav "Prompts"). List grouped by
  `agentRole`; selecting a prompt opens an editor: read-only **default** + editable
  **override** (textarea — V1, no Monaco dep), **Save** + **Reset to default**
  buttons, "modifié" badge when overridden. `lib/api.ts`: `listPrompts`,
  `savePrompt(key, content)`, `resetPrompt(key)`.

### Tests (DB-gated, throwaway DB)

- registry: `resolvePrompt` returns default when no override; returns override when
  set; returns default if `getDefault` is the only source.
- admin: `PUT` → `GET` shows `isOverridden` + content + audit row; `resolvePrompt`
  returns the new text after save (cache bust works); `DELETE` → back to default;
  `PUT` unknown key → 404; empty content → 400.

### Out of scope (YAGNI)

- Fragment-level editing (Sales' 4 fragments stay one key). Monaco editor. Tool
  toggles / model picker (the T6 spec mentions them — defer; this spec is prompts).
  Per-instance overrides (override is per role/key, process-wide).

## T10 — Team-chat (operator timeline + send-to-group)

### Goal

A `/team-chat` admin page: see the operator conversation (human-action requests +
resolutions) chronologically, and post a free-text message to the WhatsApp operator
group from the admin.

### Architecture

- **Backend** `backend/src/admin/team-chat.ts`:
  - `GET /v1/admin/team-chat?limit=` → a chronological feed derived from
    `human_actions`: each row yields a `request` entry (at `createdAt`: intent,
    severity, summary, options, correlationId) and, if resolved, a `resolved` entry
    (at `resolvedAt`: chosenOption, resolvedBy, resolvedSource). Merged + sorted desc,
    bounded by `limit` (default 50, cap 200). Plus admin-sent messages (see below)
    interleaved.
  - `POST /v1/admin/team-chat/send` `{ text }` → send to the WA operator group
    (`HUMAN_ACTION_GROUP_CHAT_ID`) via the WAHA client `sendText` + `audit_log`
    (`team_chat.message.sent`). Returns `{ ok, externalId? }`. 503 if WAHA/env not
    configured (dev). Persist the sent message (small `team_chat_messages` table, or
    reuse audit_log as the source for the "sent" entries in the feed) — **reuse
    audit_log** (query `action='team_chat.message.sent'`) to avoid a new table.
  - WAHA client + group id injected (same pattern as the reporter agent) so the test
    stubs them.
- **Frontend** `admin/src/pages/TeamChat.tsx` → `/team-chat` (nav "Équipe"):
  timeline (request/resolved/sent entries, color-coded) + a compose box (textarea +
  Send) that POSTs then refetches. Auto-refresh (e.g. 15s) or via the existing SSE.
  `lib/api.ts`: `getTeamChat`, `sendTeamChat(text)`.

### Tests (DB-gated, throwaway DB)

- timeline: seed human_actions (one pending, one resolved) → feed has the request +
  resolution entries in order with the right fields.
- send: `POST` with a stub WAHA → `sendText` called with the group id + text +
  audit row; the sent message appears in the next `GET`.

### Out of scope (YAGNI)

- Inbound WA-group message persistence / true 2-way mirror (decided against for V1).
- Threading/replies; admin→specific-agent routing (the "side-channel to Reporter"
  is just "send to group" in V1).

## Sequencing / build order

1. T6 backend: registry + migration + repo + admin endpoints + wire the ~9 prompt
   sites + tests.
2. T6 frontend: `/prompts` page + api + nav.
3. T10 backend: team-chat endpoints + tests.
4. T10 frontend: `/team-chat` page + api + nav.
5. Verify (tsc, eslint, throwaway-DB tests), commit per feature, restart backend +
   rebuild admin, live-verify, log to ruflo + update milestone.

## Risks

- T6 touches every prompt site — uniform pattern, safe fallback, but broad. The
  Sales prompt-cache interaction is the only subtlety (handled: override = one cache
  miss). Migration must run on prod `f16` (real) + the throwaway `f16_test`.
